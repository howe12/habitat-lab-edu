7-1 · 理解部署架构

① 案例含义 — 为什么用 VLN + Nav2 而不是端到端？

方法一的核心理念是分而治之：VLN 模型擅于理解指令 + 识别视觉地标（"走到厨房水池前"）， 但不擅于底层的避障和路径规划。ROS Nav2 正好相反——它能在已知地图中高效规划路径、动态避障， 但它不理解自然语言指令。

两者分工：VLN 负责"要去哪里"，Nav2 负责"怎么过去"。

② 系统架构图

三层架构：感知 → 决策 → 执行
📷 RGB 相机
/camera/rgb (sensor_msgs/Image)
📍 Odometry
/odom (nav_msgs/Odometry)
🗺️ 地图
/map (预建 2D 占据栅格)
↓   输入到   ↓
vln_inference_node   (Python)
加载 VLN checkpoint → 接收 RGB → 推理 → 输出子目标位姿 (x, y, θ)
↓   vln_ros/GoalPose   ↓
goal_publisher_node   (Python)
子目标位姿 → action 格式
→ /navigate_to_pose
↓   /navigate_to_pose (MoveToGoal Action)   ↓
ROS Nav2 Stack
Global Planner → Local Planner → Controller → /cmd_vel → 电机
↓   到达后   ↓
循环：拍新 RGB → VLN 预测下一子目标
直到 VLN 输出 STOP（到达最终目标）

节点通信拓扑

节点订阅 (Subscribe)发布 (Publish)
vln_inference_node /camera/rgb /odom /vln/subgoal_pose
goal_publisher_node /vln/subgoal_pose /navigate_to_pose (Action)
Nav2 (bt_navigator) /navigate_to_pose /cmd_vel /plan
vln_orchestrator /navigate_to_pose/result /vln/trigger_next
💡 为什么中间加一个 goal_publisher_node？

VLN 模型输出的子目标可能只是 (x, y, θ) 三个数，但 Nav2 的 NavigateToPose action 需要完整的 geometry_msgs/PoseStamped。 goal_publisher_node 专门负责格式转换 + 坐标系变换（从 VLN 的局部坐标系到 map 坐标系）， 保持 vln_inference_node 干净——它只做一件事：模型推理。

✅ 这个架构的优点

• VLN 模型和 Nav2 独立运行，一方崩溃不影响另一方
• Nav2 已有成熟的定位/规划/避障能力，不需要从头实现
• 更换 VLN 模型（如换更大的 backbone）不影响导航层
• 调试时可以单独测试每个节点的输入输出

⚠️ 这个架构的局限

• 需要预建地图（Nav2 的依赖）——不能在新环境中即开即用
• VLN 和 Nav2 的目标粒度需要调校（子目标设太远则 VLN 失去控制
  子目标设太近则效率低）
• 多了一个通信环节，增加了延迟

vln_inference_node.py — 核心推理节点

#!/usr/bin/env python3 # vln_inference_node.py — VLN 模型推理节点 # 从 /camera/rgb 读图像 + 从参数服务器读指令 → 输出子目标位姿 import rospy import numpy as np import torch from sensor_msgs.msg import Image from geometry_msgs.msg import Pose2D from cv_bridge import CvBridge class VLNInferenceNode: def __init__(self): rospy.init_node("vln_inference_node") # 加载模型 model_path = rospy.get_param("~model_path", "models/vln_inference.pth") self.model = self._load_model(model_path) self.bridge = CvBridge() self.instruction = rospy.get_param("~instruction", "") # Publisher: 输出子目标位姿 (相对坐标) self.goal_pub = rospy.Publisher("/vln/subgoal_pose", Pose2D, queue_size=1) # Subscriber: 相机图像 camera_topic = rospy.get_param("~camera_topic", "/camera/rgb/image_raw") self.img_sub = rospy.Subscriber(camera_topic, Image, self._image_cb) rospy.loginfo("VLN 推理节点已启动，等待图像...") def _load_model(self, path): # 从 Step 6 导出的 checkpoint 加载模型 # 实际使用时替换为你的 VLN Policy 类 model = torch.load(path, map_location="cpu") model.eval() return model def _image_cb(self, msg): # 收到图像 → 推理 → 发布子目标 cv_img = self.bridge.imgmsg_to_cv2(msg, "rgb8") # 预处理: resize → normalize → tensor img_tensor = torch.from_numpy(cv_img).permute(2, 0, 1).float() / 255.0 img_tensor = img_tensor.unsqueeze(0) # (1, 3, H, W) with torch.no_grad(): # 模型输出: 动作类别 (0=STOP, 1=FORWARD, 2=LEFT, 3=RIGHT) action_logits = self.model(img_tensor, self.instruction) action = torch.argmax(action_logits, dim=-1).item() # 将动作转换为子目标位姿 pose = self._action_to_pose(action) self.goal_pub.publish(pose) def _action_to_pose(self, action): # 简化版：将离散动作映射为相对位姿 # 0=STOP → 原地，1=FORWARD → 前方 1m，2=LEFT → 原地左转90° # 3=RIGHT → 原地右转90° pose = Pose2D() if action == 0: # STOP — 已到达最终目标 pose.x, pose.y, pose.theta = 0.0, 0.0, 0.0 elif action == 1: # MOVE_FORWARD pose.x, pose.y, pose.theta = 1.0, 0.0, 0.0 elif action == 2: # TURN_LEFT pose.x, pose.y, pose.theta = 0.0, 0.0, 1.5708 # 90° in rad elif action == 3: # TURN_RIGHT pose.x, pose.y, pose.theta = 0.0, 0.0, -1.5708 return pose if __name__ == "__main__": VLNInferenceNode() rospy.spin()

goal_publisher_node.py — 子目标→Nav2

#!/usr/bin/env python3 # goal_publisher_node.py — 将 VLN 输出的相对位姿转为 Nav2 Action Goal import rospy import tf2_ros import actionlib from geometry_msgs.msg import Pose2D, PoseStamped from move_base_msgs.msg import MoveBaseAction, MoveBaseGoal class GoalPublisherNode: def __init__(self): rospy.init_node("goal_publisher_node") # TF 缓存 — 用于把 VLN 输出的相对坐标转换到 map 坐标系 self.tf_buffer = tf2_ros.Buffer() self.tf_listener = tf2_ros.TransformListener(self.tf_buffer) # Nav2 / move_base Action 客户端 self.ac = actionlib.SimpleActionClient("move_base", MoveBaseAction) rospy.loginfo("等待 move_base action server...") self.ac.wait_for_server() rospy.loginfo("已连接到 move_base!") # 订阅 VLN 子目标 rospy.Subscriber("/vln/subgoal_pose", Pose2D, self._goal_cb) def _goal_cb(self, pose2d): if pose2d.x == 0 and pose2d.y == 0 and pose2d.theta == 0: rospy.loginfo("收到 STOP 信号，取消导航") self.ac.cancel_all_goals() return # 根据当前 odom 变换，将相对坐标转为绝对 map 坐标 try: trans = self.tf_buffer.lookup_transform( "map", "base_link", rospy.Time(0), rospy.Duration(1.0) ) except Exception as e: rospy.logwarn(f"TF lookup 失败: {e}") return goal = MoveBaseGoal() goal.target_pose.header.frame_id = "map" goal.target_pose.header.stamp = rospy.Time.now() # 在当前位置基础上叠加 VLN 输出的相对位移 goal.target_pose.pose.position.x = trans.transform.translation.x + pose2d.x goal.target_pose.pose.position.y = trans.transform.translation.y + pose2d.y goal.target_pose.pose.orientation.w = 1.0 # 保持当前朝向 rospy.loginfo(f"发送导航目标: ({goal.target_pose.pose.position.x:.2f}, " f"{goal.target_pose.pose.position.y:.2f})") self.ac.send_goal(goal) if __name__ == "__main__": GoalPublisherNode() rospy.spin()

💡 先 Mock 测试，再接入真实 VLN 模型

建议先用一个 mock 节点替代 vln_inference_node（例如：每 3 秒发布一个固定的子目标坐标）， 先验证整个 pipeline 的通信链路（RGB → ??? → subgoal → Nav2 → /cmd_vel → 机器人移动）可以跑通。

通信链路验证通过后，再替换为真实的 VLN 推理节点。这样可以分隔问题域——如果导航失败， 你知道是 VLN 输出的问题还是 Nav2 配置的问题。

⚠️ 适配时的关键检查项

1. 相机内参：RealSense D435 的内参与 Habitat 仿真中的针孔相机参数不同。 如果 VLN 模型在仿真中训练时使用了特定的 FOV/分辨率，真机采集的图像需要做预处理匹配（crop/resize/FOV 矫正）。
2. TF 树完整性：确保 map → odom → base_link → camera_link 链完整， 否则 goal_publisher_node 的 TF lookup 会失败。
3. Nav2 地图格式：预建地图需要与当前环境一致。如果环境布局变了，需要重新建图。
4. Python 版本兼容：ROS 1 (Python 3.8) 和 ROS 2 (Python 3.10) 的 pickle 格式兼容——确保 Step 6 导出的 checkpoint 能在两边加载。

④ 运行效果验证方法

验证步骤命令/方法预期结果
1. 节点都在 rosnode list 或 ros2 node list 看到 vln_inference, goal_publisher, move_base/bt_navigator
2. Topic 有数据 rostopic echo /vln/subgoal_pose 每收到一帧图像后，输出一个子目标位姿
3. Nav2 收到 goal 观察 rviz 中的 /plan 和 /global_plan 每次 VLN 输出新子目标时，Nav2 重新规划路径
4. 机器人移动 直接观察机器人 机器人按规划路径移动，到达子目标后停止等待下一个
5. STOP 判定 VLN 输出 STOP 后 rostopic echo /move_base/status goal 被取消，机器人停止
⚠️ 常见调试场景

场景 1："机器人不动" → 检查 rostopic echo /cmd_vel 是否有输出。 如果 /cmd_vel 为空 → Nav2 没收到 goal → 检查 goal_publisher_node 的 TF lookup 是否成功。
场景 2："机器人走错方向" → 90% 是坐标系问题。VLN 输出的坐标是相对 base_link 还是 map？ 在 rviz 中添加 Pose 显示，可视化每个子目标的位置。
场景 3："VLN 推理太慢" → 模型在 CPU 上推理可能 >500ms。考虑使用 ONNX Runtime 或 TensorRT 加速，或将模型量化到 FP16/INT8。

💡 调试建议：分阶段验证

Phase 1：不用 VLN 模型，用手动 rviz "2D Nav Goal" 验证 Nav2 基本导航正常
Phase 2：用 mock_vln_node（固定坐标输出）验证 vln → goal_publisher → Nav2 链路
Phase 3：接入真实 VLN 模型，先用 rosbag 录好的图像数据做离线推理测试
Phase 4：真机实时推理 + 导航

7-5 · 云-边-端 VLN 架构

① 案例含义

7-1 到 7-4 的架构假设 VLN 模型本地运行。但当模型大到 80 亿参数（JanusVLN）， 真机的 Jetson / 树莓派根本跑不动。7-5 介绍云-边-端分离方案： 真机负责感知+执行，云端 GPU 负责推理。

③ 核心代码

真机侧：image_bridge_node.py

import rclpy, cv2, requests, json from sensor_msgs.msg import Image from cv_bridge import CvBridge class VLNCloudBridge(rclpy.node.Node): def __init__(self): super().__init__('vln_cloud_bridge') self.bridge = CvBridge() self.cloud_url = "http://L40_IP:8000/step" self.instruction = "Go to the kitchen" # 可动态更新 # 订阅相机 self.sub = self.create_subscription( Image, '/camera/color/image_raw', self.on_image, 10) # 发布子目标给 Nav2 self.goal_pub = self.create_publisher( PoseStamped, '/goal_pose', 10) def on_image(self, msg): # 1. ROS Image → JPEG bytes cv_img = self.bridge.imgmsg_to_cv2(msg, 'bgr8') _, jpg = cv2.imencode('.jpg', cv_img, [cv2.IMWRITE_JPEG_QUALITY, 80]) # 2. 上传到云端 resp = requests.post(self.cloud_url, files={'image': jpg.tobytes()}, data={'instruction': self.instruction}, timeout=5) # 3. 解析动作 action = resp.json()['action'] # "MOVE_FORWARD" self.get_logger().info(f'Action: {action}') # 4. 映射为 Nav2 目标并发布 if action == 'STOP': self.cancel_nav() else: goal = self.action_to_goal(action) self.goal_pub.publish(goal)

④ 每步延迟预算

阶段耗时占比优化空间
拍照 + 压缩~0.05s2%已足够
网络上传 (50KB)~0.02s1%WiFi 6 / 5G 无压力
JanusVLN 推理~2.0s80%模型量化 / TensorRT
网络回传~0.01s0%可忽略
Nav2 规划~0.1s4%已足够
机器人移动 (0.25m)~0.3s13%取决于速度
合计~2.5 秒/步
⚠️ 推理速度是瓶颈——但不是死结

2 秒/步听起来慢，但 VLN 导航通常 10-30 步到达目标 → 总耗时 25-75 秒，实际可用。 如果接受不了，可以：模型量化到 INT8（速度 2x）、换小模型（Qwen2.5-VL-3B）、 或等下一代 GPU（RTX 5090 推理 <0.5s）。

7-6 · 仿真与真机测试计划

① 案例含义

直接上真机风险大（撞墙、电机过载）。7-6 设计了两阶段测试计划：先在 Gazebo 仿真 中验证完整链路， 再上 Spark-I 真机。每阶段都有明确的 Pass/Fail 条件。

② Phase 1：Gazebo 仿真测试

环境：Spark-I Gazebo 仿真 + TurtleBot3 世界

#测试项操作预期Pass 条件
T1 通信链路 真机端启动 image_bridge → curl 云端 /health 200 OK 3/3 请求成功
T2 单步推理 发一张仿真 RGB + "Go to the door" 返回一个有效动作 动作在 {FORWARD,LEFT,RIGHT,STOP} 内
T3 动作映射 FORWARD → 计算目标位姿 → 检查合理性 位姿在机器人前方 0.25m 位置误差 < 0.05m，角度误差 < 5°
T4 Nav2 单步导航 发布一个 goal_pose → 观察机器人移动 机器人到达目标点 最终位置距目标 < 0.1m
T5 多步导航 连续 10 步 VLN → Nav2 循环 每步正确执行 10/10 步无卡死、无超时
T6 完整 episode 发指令 → 自动运行直到 STOP 机器人停在目标附近 最终位置距 ground-truth goal < 2m
T7 异常恢复 推理超时 / 网络断开 → 观察行为 安全停止，不发无效指令 机器人不失控移动

#测试项特殊注意事项Pass 条件
T8 相机标定验证 真机 RGB 尺寸/视场角与仿真不同 图像 resize 后送入模型无报错
T9 坐标系对齐 VLN 输出是机器人坐标系，需转 map 系 rviz 中 goal_pose 出现在正确位置
T10 实景单步导航 真机有里程计漂移，Nav2 需实时修正 单步到达误差 < 0.15m
T11 动态障碍物 人在机器人前方走过 → Nav2 应避障 机器人减速/绕行，不撞人
T12 10 条指令测试集 在办公室环境中预设 10 条导航指令 SR ≥ 40%（10 条中 4+ 成功）
T13 紧急停止 物理急停按钮或 /vln/stop service 电机立即断电，< 0.5s 响应
⚠️ 真机测试安全守则

1. 首次测试用最低速度（0.1 m/s），逐步提高
2. 旁边始终有人手持急停遥控器
3. 测试区域清空行人，地面无杂物
4. 先测试 STOP 指令（验证急停链路），再测移动
5. 录 rosbag，失败后可回放分析
