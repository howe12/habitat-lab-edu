关键洞察：从几何到语义的谱系
PointNav → ObjectNav → ImageNav → VLN 的演进反映了 Embodied AI 的核心挑战：
· PointNav = "走到 (x,y)" — 纯几何，GPS/Compass 告诉你相对位置 → 最简单的 RL 训练
· ObjectNav = "找到 chair" — 需要语义理解（chair 长什么样？它通常在哪？）
· ImageNav = "到达这里" — 需要视觉匹配（当前视野 vs 目标图片）→ 没有 GPS
· InstanceImageNav = "找到这把特定的椅子" — 需要实例级识别（区分同类的不同个体）
· VLN = "按指令去卧室" — 需要语言理解 + 视觉匹配 + 路径记忆 → 最接近真实机器人

1.2 PointNav — 几何坐标导航 难度 ⭐⭐

① 案例含义
PointNav（PointGoal Navigation）是 Habitat 中最基础、最成熟的导航任务。 智能体接收一个相对目标坐标（距离 ρ 和角度 φ），通过 RGB-D 视觉和运动控制， 自主规划路径走到目标点。这是 Embodied AI 的 "Hello World"。
② 核心配置与关键调用
配置项值说明
任务类型Nav-v0habitat/tasks/nav/nav.py 中注册的 NavigationTask
任务配置config/habitat/task/pointnav.yaml定义 actions, measurements, sensors
目标传感器pointgoal_with_gps_compass输出 (ρ, φ) — 当前 agent 相对目标的距离和角度
基准配置config/benchmark/nav/pointnav/5 种场景：gibson, mp3d, hm3d, habitat_test, franka
训练配置habitat_baselines/config/pointnav/ppo_pointnav.yaml / ddppo_pointnav.yaml
动作空间4 个离散动作STOP(0), MOVE_FORWARD(1), TURN_LEFT(2), TURN_RIGHT(3)
观测空间RGB(256×256×3), Depth(256×256×1),
PointGoal(2,)		视觉为 256×256 分辨率（可调）
 
# 用 Gym API 创建 PointNav 环境
import gym, habitat.gym  # noqa: F401 — 注册 Habitat-v0

env = gym.make("Habitat-v0",
    cfg_file_path="benchmark/nav/pointnav/pointnav_habitat_test.yaml",
)

obs = env.reset()
print(obs.keys())
# dict_keys(['rgb', 'depth', 'pointgoal_with_gps_compass'])

# pointgoal_with_gps_compass = [ρ, φ]
# ρ = 距离目标（米），φ = 相对目标的偏航角（弧度）
pg = obs["pointgoal_with_gps_compass"]
print(f"距离目标: {pg[0]:.2f}m, 偏航角: {pg[1]:.2f}rad")

# 执行一个随机动作
obs, reward, done, info = env.step(env.action_space.sample())
env.close()

$ python -u -m habitat_baselines.run \
    --config-name=pointnav/ppo_pointnav_example.yaml

# PointNav 评估
$ python -u -m habitat_baselines.run \
    --config-name=pointnav/ppo_pointnav_example.yaml \
    habitat_baselines.evaluate=True

# 快速运行 Demo（环境探索 + 训练 + 推理）
$ python examples/rl_nav_demo.py --train-steps 50000
④ 输出结果含义
1. pointgoal_with_gps_compass: 这是一个 2 维向量 [ρ, φ]，ρ 是当前 agent 到目标位置的距离（米），φ 是相对偏航角（弧度）
2. φ 的正负号: 正值 = 目标在 agent 右侧，负值 = 目标在左侧
3. distance_to_goal: 测地距离（沿可行走表面的最短路径），不是欧氏距离
4. GPS+Compass 是特权信息: 真实机器人没有完美的 GPS，需要用 SLAM 或视觉里程计来近似
5. reward = -(新距离 - 旧距离): 靠近目标得正奖励，远离目标得负奖励（distance_to_goal_reward 计算）
⑥ 试试调整这些
调整项怎么改预期看到什么
max_episode_steps 在 benchmark yaml 中改为 100 或 1000 步数限制影响成功率——短限制迫使策略更高效
success_distance 修改 task.measurements.success.success_distance 默认 0.2m；设为 0.1 更严格（更难），设为 0.5 更宽松
RGB 分辨率 修改 rgb_sensor width/height 参数 分辨率越高视觉信息越多，但训练更慢

1.3 ObjectNav — 物体类别导航 难度 ⭐⭐⭐

① 案例含义
ObjectNav（ObjectGoal Navigation）要求智能体在未知环境中找到 指定类别的物体。 与 PointNav 不同，ObjectNav 不给你目标坐标，只告诉你目标物体类别（如 "chair"、"bed"）， 智能体必须自主探索、识别语义，在找到目标物体后决定 STOP。
② 核心配置与关键调用
配置项值说明
任务类型ObjectNav-v1object_nav_task.py → @registry.register_task(name="ObjectNav-v1")
任务配置config/habitat/task/objectnav.yaml6 个动作，GPS+Compass+objectgoal 传感器
目标传感器objectgoal_sensor输出目标物体类别 ID（整数索引）
辅助传感器gps_sensor + compass_sensor提供绝对位置/朝向（帮助定位）
额外动作look_up + look_down上下看（±30°）→ 物体可能在高处或低处
数据集HM3D / MP3D / HSSD / ProcTHOR每类物体有 6 种（如 6 种椅子），增加泛化难度
距离指标distance_to: VIEW_POINTS使用视角点距离（能看到物体的位置），非物体中心距离
训练配置config/objectnav/ddppo_objectnav_hm3d.yaml训练步数可达 2.7 亿步（270000 updates × 1000 steps）
特殊指标soft_spl即使未 STOP 在物体旁，走到物体附近也有部分分数
 
import gym, habitat.gym

# ObjectNav 需要下载 HM3D 数据集（或使用 HSSD）
env = gym.make("Habitat-v0",
    cfg_file_path="benchmark/nav/objectnav/objectnav_hm3d.yaml",
)

obs = env.reset()
print(obs.keys())
# dict_keys(['rgb', 'depth', 'objectgoal', 'gps', 'compass'])

# objectgoal = 目标物体类别的整数 ID
print(f"Target object ID: {obs['objectgoal'].item()}")

# ObjectNav 有 6 个动作: STOP, MOVE_FORWARD, TURN_LEFT,
#                            TURN_RIGHT, LOOK_UP, LOOK_DOWN
print(f"Action space: {env.action_space}")
# Discrete(6)

# 执行一个 episode: 随机动作直到 done
total_reward = 0.0
for step in range(500):
    action = env.action_space.sample()
    obs, reward, done, info = env.step(action)
    total_reward += reward
    if done:
        print(f"Episode done at step {step+1}, 
              f"success={info.get('success', 0)}, 
              f"distance={info.get('distance_to_goal', '?')}")
        break
env.close()
③ 运行命令
# ObjectNav DD-PPO 训练（需要 HM3D 数据集）
$ python -u -m habitat_baselines.run \
    --config-name=objectnav/ddppo_objectnav_hm3d.yaml

# ObjectNav 评估
$ python -u -m habitat_baselines.run \
    --config-name=objectnav/ddppo_objectnav_hm3d.yaml \
    habitat_baselines.evaluate=True \
    habitat_baselines.eval_ckpt_path_dir=/path/to/checkpoints \
    habitat_baselines.checkpoint_folder=/path/to/checkpoints
④ 输出结果含义
1. objectgoal: 一个整数，表示目标物体类别的索引。HM3D 定义了 6 种物体类别（chair, bed, plant, toilet, tv_monitor, sofa）
2. gps: 2 维向量 [x, y]，agent 在场景中的绝对坐标。注意这是 绝对 GPS，不是 PointNav 中的相对坐标
3. compass: [sin(θ), cos(θ)]，agent 当前的朝向角度
4. VIEW_POINTS 距离: 计算的是 agent 到能看到目标物体的 最近视点 的距离，而非物体本身的位置（物体位置对 agent 不可见）
5. soft_spl: SPL 的 "软" 版本——即使 agent 没有在物体前 STOP，只要它走到了物体附近（按距离分等级），也能得部分分
ObjectNav 为什么比 PointNav 难得多？
1. 无目标坐标: 你只知道要找 "chair"，不知道它在哪。必须用视觉探索。
2. 语义理解: CNN 需要学习 chair 视觉特征 vs 其他物体。
3. 泛化压力: HM3D 有 6 种椅子，测试时可能出现完全不同的椅子。
4. 探索-利用: 是继续搜索还是在找到疑似目标时 STOP？这是一个赌博。
⑥ 试试调整这些
调整项怎么改预期看到什么
数据集 切换到 benchmark/nav/objectnav/objectnav_hssd-hab.yaml HSSD 数据集 vs HM3D，物体分布和场景风格不同
success_distance 修改 task.measurements.success.success_distance 默认 0.1m；ObjectNav 用更严的标准（必须在物体旁 STOP）
物体类别 运行物体类别统计脚本 看哪些类别的物体更难找（被遮挡、在角落）

1.4 ImageNav — 图片目标导航 难度 ⭐⭐⭐

① 案例含义
ImageNav（ImageGoal Navigation）给智能体一张目标图片（从目标位置看到的 RGB 画面）， 智能体必须在到达目标位置时 STOP。智能体没有 GPS 信号，只能靠视觉匹配完成导航。 这更接近人类导航方式——"到达这个地方（给你看照片）"。
ImageNav vs PointNav 的关键区别
· PointNav: 给你 GPS 坐标 → "目标在你的东北方向 5.2 米"
· ImageNav: 给你 一张照片 → "目标长这样，你来找"
→ ImageNav 没有 GPS 传感器！策略必须通过视觉匹配来逼近目标位置。 ImageNav 虽然复用 PointNav 数据集（在目标位置渲染图片），但训练难度远高于 PointNav。
② 核心配置与关键调用
配置项值说明
任务类型Nav-v0复用 PointNav 的基础任务类
任务配置config/habitat/task/imagenav.yaml4 个动作，只有 imagegoal 传感器
目标传感器imagegoal_sensor输出目标位置的 RGB 图像
基准配置config/benchmark/nav/imagenav/imagenav_gibson.yaml / imagenav_mp3d.yaml / imagenav_test.yaml
训练配置habitat_baselines/config/imagenav/ddppo_imagenav_gibson.yaml (ResNet50 视觉编码器)
数据集共用 PointNav 数据集运行时在目标位置渲染 RGB 作为 goal image
动作空间4 个离散动作同 PointNav（STOP/MOVE_FWD/TURN_LEFT/TURN_RIGHT）
观测空间RGB + Depth + 目标 RGB 图像目标图像和当前观测的 RGB 需要被联合编码比较
 
import gym, habitat.gym

env = gym.make("Habitat-v0",
    cfg_file_path="benchmark/nav/imagenav/imagenav_mp3d.yaml",
)

obs = env.reset()
print(obs.keys())
# dict_keys(['rgb', 'depth', 'imagegoal'])

# imagegoal = 目标位置看到的 RGB 图片
goal_img = obs["imagegoal"]
print(f"Goal image shape: {goal_img.shape}")  # (H, W, 3)

# 注意: 没有 GPS/Compass 传感器！
# 策略唯一知道的就是目标图片 + 当前视觉观测
env.close()

$ python -u -m habitat_baselines.run \
    --config-name=imagenav/ddppo_imagenav_gibson.yaml

# ImageNav 评估
$ python -u -m habitat_baselines.run \
    --config-name=imagenav/ddppo_imagenav_gibson.yaml \

ImageNav 的独特挑战
1. 视觉歧义: 很多位置看起来很像（走廊两端、相似房间）→ 容易走过头
2. 视点变化: 你从西边走向目标，目标图片是从东边拍的 → 视野完全不同，无法直接匹配
3. 无定位传感器: 没有任何坐标信号 → 策略必须在脑中建立隐式空间认知
4. 策略设计: 通常需要 Siamese 网络 或 视觉对比编码器 来比较当前观测和目标图片

1.5 InstanceImageNav — 实例级图像导航 难度 ⭐⭐⭐⭐

① 案例含义
InstanceImageNav 是 ImageNav 的进阶版：不是找 "任意椅子"，而是找 特定的那把椅子。 给你一张 目标实例的图片，你需要找到场景中 特定的物体实例。 这比 ObjectNav 更难——你必须区分 "这把棕色皮椅" vs "那把黑色布椅"，而不是笼统地找 "椅子"。

1.6 VLN — 视觉语言导航 难度 ⭐⭐⭐⭐⭐

① 案例含义
VLN（Vision-and-Language Navigation）要求智能体根据 自然语言指令 在未知环境中导航到目标。这是 5 种任务中最具挑战性的：需要 语言理解 + 视觉感知 + 决策 三者紧密结合。典型指令如 "Go down the hallway, turn left at the bathroom, and stop in front of the bed."
② 核心配置与关键调用
配置项值说明
任务类型VLN-v0nav.py 中注册的 VLN 任务变体
任务配置config/habitat/task/vln_r2r.yaml4 个动作，instruction_sensor
目标传感器instruction_sensor输出自然语言指令文本
基准配置config/benchmark/nav/vln_r2r.yamlMP3D 的 R2R 数据集
数据集R2R (Room-to-Room) — MP3D 场景人类标注的路径+指令对，训练/验证/测试 分集
动作空间4 个离散动作STOP/MOVE_FWD(0.25m)/TURN_LEFT(15°)/TURN_RIGHT(15°)
关键指标nDTW (normalized Dynamic Time Warping)衡量 agent 实际路径与参考路径的相似度
RL 训练支持⚠️ 无官方 PPO 配置baselines 中没有 VLN 的 RL 训练配置，主要用 Imitation Learning
 
import gym, habitat.gym

env = gym.make("Habitat-v0",
    cfg_file_path="benchmark/nav/vln_r2r.yaml",
)

obs = env.reset()
print(obs.keys())
# dict_keys(['rgb', 'depth', 'instruction'])

# instruction = 自然语言导航指令
instr = obs["instruction"]
print(f"Instruction: {instr}")
# 例如: "Go down the hallway past the bathroom on your left.
#         Stop when you reach the bedroom on the right."

# VLN 的核心挑战: 将自然语言转化为时空动作序列
env.close()
③ 运行命令
# VLN 评估（使用 imitation learning 训练的模型）
# 注意: baselines 中没有 VLN 的 PPO 训练配置
# VLN 主要通过 Imitation Learning (DAgger / BC) 训练
$ python -u -m habitat_baselines.run \
    --config-name=eqa/il_pacman_nav.yaml
VLN 的独特挑战与局限
1. 语言接地 (Language Grounding): "turn left at the bathroom" → 必须先识别 bathroom，再决定在哪里转弯
2. 时序推理: 指令是时序的："先走到走廊尽头，然后右转" → 必须按顺序执行
3. 部分可观测性: 看不到的物体需要记忆或推理
5. 评估复杂: 除了 success/SPL，还需要 nDTW 来衡量路径与参考路径的相似度
⚠️ VLN 在这个工程中不是用 RL 训练的
Habitat-Baselines 的 PPO Trainer 硬编码了 supported_tasks = ["Nav-v0"]， 而 VLN 注册为 VLN-v0 → PPO 无法直接用于 VLN。 VLN 使用 Imitation Learning（DAgger / 行为克隆）而非 RL。 为什么？因为语言指令导航的奖励函数极难定义——"在浴室处左转" 走偏了， 无法用简单的距离差来衡量。学术界有少量 RL+IL 混合方案，但 Habitat-Baselines 未收录。

1.7 选择指南：用哪种任务？

你的目标推荐任务理由
学习 RL 导航基础 PointNav 数据离线可用、训练快、文档最全、Demo 开箱即用
研究语义视觉搜索 ObjectNav 需要识别物体类别、有多种数据集和 baseline
研究纯视觉导航（无 GPS） ImageNav 只能用视觉匹配到达目标，是最纯粹的视觉导航
研究细粒度实例识别 InstanceImageNav 区分同类的不同个体，但训练成本极高
研究语言+视觉+导航融合 VLN 语言指令导航，但 RL 训练支持有限，需自行搭建

1.7 问答导航：EQA (Embodied Question Answering)

EQA 是唯一需要语言理解 + 视觉 + 导航三者融合的任务。Agent 在 MP3D 场景中导航，被问到关于场景的问题 （如 "What color is the car in the garage?"），需要找到目标、观察、然后用自然语言回答。 注册为 EQA-v0，使用 MP3DEQA-v1 数据集（819 个问答对）。

组件类型说明
QuestionSensorlab sensor返回问题文本的 token ID 序列（变长序列），观测量类型为 TOKEN_IDS
AnswerActiontask action从答案词汇表选择一个词作为最终回答，action_args = {"answer_id": int}

EQA 任务的基本使用
import habitat
from habitat.tasks.eqa.eqa import AnswerAction

config = habitat.get_config("benchmark/nav/eqa_mp3d.yaml")
env = habitat.Env(config)
obs = env.reset()
# obs 包含: rgb, depth, semantic, question（token IDs 列表）
print("Question:", env.current_episode.question.question_text)
print("Question tokens:", obs["question"])

# 导航若干步后，提交答案
correct_id = env.current_episode.question.answer_token
env.step({"action": AnswerAction.name, "action_args": {"answer_id": correct_id}})
print(env.get_metrics())  # answer_accuracy = 1.0
EQA 与其他导航任务的关键区别

EQA 的 episode 不会在到达导航目标时自动终止 — agent 必须主动发出 AnswerAction 才会结束。 如果 agent 只导航不回答，会一直运行直到 max_episode_steps（默认 500）。 这也是为什么 EQA 比 PointNav/ObjectNav 更难：agent 需要学会何时停止导航并开始回答。

2. Policy 网络结构

PointNavResNetPolicy — Habitat 中 PointNav 任务的标准策略网络。 理解网络的每一层，才能回答核心问题：RL 到底在训练什么？

2.1 网络全景：CNN + RNN + Actor/Critic

📷 Visual Encoder ResNet18/50 → 压缩空间特征
+
🧭 GPS+Compass PointGoal 传感器 (2维向量)
↓ 拼接 → Linear(hidden_size)
🔄 GRU (可选) hidden_size=512, num_layers=1
↓ 隐藏状态
🎯 Actor Head Linear → Categorical(4 actions)
💰 Critic Head Linear → 1 (value scalar)
💡 为什么要 GPS+Compass？
PointNav 任务中，agent 不仅有视觉输入（RGB/Depth），还有 pointgoal_with_gps_compass 传感器提供的 目标方向+距离（2维向量）。这是一个关键的"特权信息"——没有它， 纯视觉导航会难得多。策略网络将视觉特征和 GPS 向量拼接后一起送入决策头。

2.2 "RL 到底训练的是什么？"

一句话回答
训练的是整个策略网络的权重参数——不是规则、不是地图、不是规划器，而是一个从像素到动作的端到端映射函数。
模块参数量 (ResNet18)学的是什么类比
视觉编码器
ResNet18/50		 ~11M 从 RGB-D 像素中提取空间特征：障碍物在哪、房间结构、物体纹理 "眼睛"——把像素变成有意义的结构信息
RNN 状态编码器
GRU/LSTM 512维		 ~3M 记忆历史观测，形成隐式空间认知：走过的路、看过的角落 "海马体"——在脑中维持对空间的动态记忆
Actor 头
Linear → Categorical		 ~200K 给定当前状态，输出4 个动作的概率分布——哪个动作最可能到达目标 "决策者"——看情况选动作
Critic 头
Linear → 1		 ~100K 估计当前状态的未来预期回报（value）——这个位置有利吗？ "评论家"——好坏我来评判
训练更新的是权重，不是行为规则。 对于一张包含走廊和椅子的 RGB 图像，训练后的网络学会给它分配低 value（远离目标） 并降低 TURN_LEFT 概率（因为左边是墙）。但这些知识不是被编程进去的——是 CNN 卷积核 通过 PPO 梯度更新从数百万步交互中自己浮现出来的。

2.3 "RL 需要基础模型吗？是微调吗？"

取决于配置——有三种模式
Habitat RL 的支持介于"从零训练"和"预训练迁移"之间，你可以根据任务选择。
模式配置开关说明类比
A — 从零训练 SimpleCNN 默认 所有参数随机初始化，完全从零学。适合小规模实验，收敛慢 训练一个全新的 CNN 做图像分类
B — 预训练编码器 pretrained_encoder: True
pretrained_weights: gibson-2plus-resnet50.pth		 从 Gibson 数据集 DD-PPO 训练的 ResNet50 加载视觉编码器权重，其余随机初始化，然后在新数据集（HM3D/MP3D）上继续 PPO NLP 中 "加载 BERT 权重，在新的下游任务上微调"
C — CLIP 特征提取 backbone: resnet50_clip_avgpool 加载 OpenAI CLIP ResNet50，视觉编码器完全冻结（requires_grad=False）。策略只学如何利用 CLIP 语义特征导航 用 GPT 的 embedding 做情感分类——特征提取器不变，只用它的输出
Habitat 的 "微调" 与 NLP 的微调有何不同？
NLP (BERT → 分类)Habitat RL (Gibson → HM3D)
预训练来源海量文本自监督 (MLM)DD-PPO 在 Gibson 上的 RL 训练
预训练规模百亿 token数亿步交互
迁移什么语言理解能力视觉特征提取能力
微调方式加分类头 + 小 LR 全参数更新冻结/解冻编码器 + 继续 PPO 更新
本质任务迁移（语言 → 情感分类）领域迁移（Gibson 场景 → HM3D 场景）

2.4 "训练好的模型和环境、设备有关吗？"

因素相关程度说明缓解措施
场景几何 高 Gibson 训练的模型直接跑 MP3D 测试，成功率可掉 20%+——房间布局、走廊宽度不同 预训练编码器 + 目标数据集微调
传感器分辨率 高 256×256 训练的模型接收 640×480 输入会出错（FC 层输入维度固定） resize 预处理；或用全卷积网络
动作空间 高 4-action (PointNav) 模型不能直接用于 6-action (ObjectNav) 任务——Actor 头维度不同 重新训练 Actor/Critic 头
RGB vs 深度 中 有深度的模型迁移到纯 RGB 场景性能下降——深度提供几何信息 多任务训练；深度 dropout
数据集 split 中 同一场景的不同 episode（train vs val）泛化通常较好；不同场景则考验泛化能力 严格按照 train/val/test 分离
GPU 设备 低 PyTorch 自动处理 CUDA/CPU 转换；checkpoint 可跨 GPU 加载（map_location） 无需特殊处理
Habitat-Sim 版本 低 传感器输出格式跨版本兼容；极少情况需要重新导出 checkpoint 固定版本依赖
本质：Habitat RL 学的是视觉运动策略，不是场景地图。 策略编码的是 "看到这样的走廊，应该这样做" 的通用能力，而不是某个特定房间的布局。 这就是为什么可以通过预训练编码器跨数据集迁移——视觉特征提取的底层卷积核（边缘、纹理、形状） 在不同场景间是可以共享的。

3. RL 环境包装链

从最内层的 Env 到最外层的 VectorEnv，每一层封装增加一个能力维度。

VectorEnv (N 个并行) habitat/core/vector_env.py
↑ 管理 N 个独立环境实例，合并 batch
GymHabitatEnv habitat/gym/gym_definitions.py
↑ 将字典 action/obs 转换为 gym 标准空间
RLEnv (gym.Env) habitat/core/env.py:357
↑ step() → (obs, reward, done, info)
Env (核心) habitat/core/env.py:39
↑ 组合 Simulator + Task + Dataset
🏗 Simulator habitat_sim 渲染引擎
🧠 Task (Nav-v0) reward + success + measurements
📦 Dataset episodes 迭代器
💡 RLEnv 做了什么？
原始 Env.step() 返回的是 Observations（字典）。
RLEnv.step() 包装后返回标准的 (obs, reward, done, info) 元组：
· reward 来自 task.measurements 中的 reward_measure（如 distance_to_goal_reward）
· done 由 task 判断 episode 是否结束（到达目标 / 超步数 / 碰撞）
· info 包含 task.get_metrics() 返回的所有指标

4. PPO 训练算法

Proximal Policy Optimization (Schulman et al., 2017) — Habitat 使用的核心 RL 算法。

4.1 PPO 训练循环 (PPOTrainer.train)

① 初始化 创建 VectorEnv → Policy → Updater
↓
② 收集 Rollout N 个 env × num_steps 步
→
③ 计算 Returns + Advantages GAE (gamma=0.99, tau=0.95)
↓
④ PPO Update ppo_epoch=1, num_mini_batch=1
→
⑤ Clip Gradient + Step clip_param=0.1, max_grad_norm=0.5
↓
⑥ Log + Checkpoint tensorboard / 定期保存
⟲
重复 ②-⑥ 直到 total_num_steps

4.2 PPO 核心参数详解

clip_param = 0.1

默认 0.2，示例中用 0.1
PPO 的核心创新——限制策略更新的幅度。新策略与旧策略的概率比被裁剪到 [1-ε, 1+ε]。 越小越保守，训练更稳定但收敛更慢。

lr = 2.5e-4

Adam 优化器学习率
标准设置。可配合 use_linear_lr_decay=True 让学习率从初始值线性衰减到 0。

num_steps = 32

每次 rollout 收集的环境步数
每次 PPO 更新前，每个环境执行 32 步。batch_size = num_environments × num_steps。 更大的值提供更多样本但降低更新频率。

gamma = 0.99

折扣因子
未来 reward 的衰减系数。0.99 意味着 100 步后的 reward 权重约为 0.99^100 ≈ 0.37。 接近 1.0 表示重视远期奖励。

tau = 0.95 (GAE λ)

Generalized Advantage Estimation
控制 advantage 估计中偏差-方差的权衡。λ=1 等价于 Monte Carlo（无偏但高方差）； λ=0 等价于 TD(0)（有偏但低方差）。0.95 是经验最优值。

entropy_coef = 0.01

熵正则化系数
鼓励策略保持探索（不收敛到单一动作）。值越大探索越多，可设置 use_adaptive_entropy_pen=True 自动调整。

4.3 DD-PPO (分布式 PPO)

当单卡不够时，使用 DD-PPO 在多 GPU/多节点上训练。它与 PPO 共享同一个 PPOTrainer 类， 区别在于通过 DecentralizedDistributedMixin 增加了分布式同步。

sync_frac = 0.6

每 60% 步数后同步一次梯度
不每步同步（那样通信开销太大），而是积累一定步数后平均梯度。

backbone = "resnet18"

视觉编码器 (ResNet18/50/101)
DD-PPO 默认使用 ResNet18 作为视觉 backbone。支持 pretrained_encoder=True 加载预训练权重。

num_recurrent_layers = 1

GRU 循环层数
Policy 在 CNN 特征后接 GRU 层处理序列信息。设为 0 可禁用循环层。

distrib_backend = "GLOO"

PyTorch 分布式后端
CPU 训练用 GLOO，GPU 训练可用 NCCL（更快）。

6. RL 导航实战：观测→动作→奖励

在深入 PPO 算法之前，先用一个完整的 Demo 理解 RL 如何让 agent 学会导航。 观测什么、输出什么、奖励什么——这三个问题是一切 RL 训练的基础。

6.1 传统导航 vs RL 导航

RL 导航与传统 ROS Nav2 的根本区别：不分解为独立模块，而是端到端学习。

传统导航 (ROS Nav2) SLAM 建图 → AMCL 定位 → A* 规划 → DWA 控制
vs
RL 导航 (Habitat PointNav) RGB+Depth+GPS → CNN+GRU → (前进|左转|右转|停止)
🔑 关键差异
RL 用 CNN+GRU 隐状态 替代了显式地图和规划器。 GPS+Compass 传感器提供 完美的相对目标向量（距离+角度）， 这是一种特权信息——它绕过了 SLAM 问题，让策略专注学习"如何从视觉走到目标"。 真实机器人需要用视觉里程计或 SLAM 来近似这个信号。

# examples/rl_nav_demo.py — Part 1（简化版） import gym import habitat.gym # 注册 Habitat-v0 env = gym.make("Habitat-v0", cfg_file_path="benchmark/nav/pointnav/pointnav_habitat_test.yaml") # 观测空间 obs = env.reset() # obs = {'rgb': (256,256,3), 'depth': (256,256,1), # 'pointgoal_with_gps_compass': (2,)} # 动作空间: 0=STOP, 1=MOVE_FORWARD, 2=TURN_LEFT, 3=TURN_RIGHT action = env.action_space.sample() # 随机动作 obs, reward, done, info = env.step(action) # reward = -(新距离 - 旧距离)，靠近目标为正

④ 运行效果

⑤ 输出结果的含义
  1. rgb / depth — agent 第一视角的视觉输入。RGB 是 0-255 的 uint8 像素，depth 是浮点米值（0-10m）。
  2. pointgoal_with_gps_compass — [距离 ρ, 角度 φ]。ρ 是 agent 到目标的直线距离（米），φ 是相对偏转角（弧度）。这是 RL 的关键"特权信息"。
  3. STOP/MOVE_FORWARD/TURN_LEFT/TURN_RIGHT — 离散动作空间，每次只能选一个。turn_angle=10°，意味着转 90° 需要 9 次 TURN 动作。
  4. reward = -(新距离 - 旧距离) — 靠近目标得正奖励，远离得负奖励。0.367 的总奖励意味着 agent 在 7 步随机移动中总体靠近了目标约 0.37m。
  5. success=0.0, SPL=0.0 — 随机 agent 不可能导航成功，这是 baseline。
⑥ 调整项怎么改预期看到什么
增大探索步数修改 range(50) → range(200)更长的随机游走，可能看到更多场景区域

打印每一步的 goal 向量step 循环里加 print(obs["pointgoal_with_gps_compass"])看到目标距离在不断变化（随机动作时增减不定）

💡 为什么 Demo 里策略只训练了 5000 步？
默认 --train-steps 50000 只够验证 pipeline 是否正常。 要让策略真正学会导航，需要 数百万甚至上亿步（生产配置用 total_num_steps=7.5e7）。 因此 Part 4 输出的 3 个 episode 都会看到 [FAILED] 和 1 步 STOP——策略还没学会如何移动。 完整训练后再跑推理，才能看到有效的导航行为。

7. 评估与 Benchmark

训练完成后，使用 habitat.Benchmark 或 trainer 的 eval() 评估模型性能。

7.1 Benchmark 评估流程

# 方式 1：直接使用 Benchmark from habitat.config.default import get_config import habitat cfg = get_config('benchmark/nav/pointnav/pointnav_habitat_test.yaml') benchmark = habitat.Benchmark(cfg) metrics = benchmark.evaluate(agent, num_episodes=100) # → {'success': 0.42, 'spl': 0.35, 'distance_to_goal': 1.23, ...}

7.2 评估指标详解

指标全称范围含义
success Success Rate [0, 1] Agent 是否在 max_episode_steps 内到达目标（距离 < success_distance）
spl Success weighted by (normalized inverse) Path Length [0, 1] 成功率 × (最短路径长度 / max(实际路径长度, 最短路径长度))。惩罚绕路
distance_to_goal Final Distance to Goal [0, ∞) m Episode 结束时 agent 到目标的欧式距离。越小越好
num_steps Episode Steps Taken [0, max_steps] Agent 在 episode 中执行的步数（非 STOP 动作次数）
collisions Collision Count [0, ∞) Agent 碰撞墙壁/障碍物的次数（previous_step_collided）
💡
SPL vs Success：Success 只关心"到了没有"，SPL 关心"走了多高效"。 两个 agent 可能都有 80% success，但 SPL 更高的那个路径更短。 实际项目中 SPL 往往是更重要的指标，因为它反映了导航效率。

7.3 使用 Trainer 评估已有 Checkpoint

# 方式 2：命令行评估（通过 run.py） # 在 habitat-baselines 目录下运行： python -u -m habitat_baselines.run \ --config-name=pointnav/ppo_pointnav_example \ habitat_baselines.evaluate=True \ habitat_baselines.eval_ckpt_path_dir="data/new_checkpoints/ckpt.50.pth" \ habitat_baselines.eval.video_option='["disk"]'

设置 eval.video_option: ["disk"] 会在评估时保存 agent 的第一视角视频到 video_dir。 设置 ["tensorboard"] 会将视频上传到 TensorBoard。 视频对于调试和理解 agent 行为非常有帮助。

7.4 训练 vs 评估的数据集分离

Train Split 用于训练 RL policy
Val Split 用于调超参 + 早停
Test Split 最终评测（不用于训练）

在配置文件中通过 habitat.dataset.split: "train" 指定使用哪个 split。 训练时用 train split，评估时用 val/test split。这是 ML 的基本实践但在 RL 中容易被忽略。

8.4 常见问题排查

现象可能原因解决方法
reward 不上升 学习率过高/过低、reward 缩放不正确 调整 lr (试 1e-4 ~ 1e-3)，检查 reward_measure 定义

10. HRL 分层强化学习

Hierarchical RL 将复杂长周期任务分解为两级策略：High-Level (HL) 决策层选择技能 → Low-Level (LL) 技能层执行。 适用于 Rearrange 等需要多阶段操作的任务。

HRL 两层架构

HL Policy（高层策略）：观察环境全局状态 → 从 7 种技能中选择一个 → 传递给 LL 层执行。 HL 可以是 FixedHighLevelPolicy（按 PDDL solution 脚本化执行，无需训练）或 NeuralHighLevelPolicy（ResNet+LSTM 编码器 → CategoricalNet，需要训练）。

LL Skills（低层技能）：每个 Skill 是一个独立的策略模块，执行固定数量的 step 直到满足终止条件 （例如物体被抓起、到达导航目标）或超时。控制权在技能终止后归还 HL 层。

关键设计：loss 只在 HL 决策步计算（非每步），实现长周期信用分配。这称为 Options Framework / Semi-MDP。

技能名称作用说明
Nav导航到目标物体基于预训练 Navigation policy 的 NN 技能，由 PDDL action 参数指定目标
Pick抓取物体控制 Fetch 手臂靠近物体 → 激活吸盘抓取 → 抬起手臂
Place放置物体将持有的物体移动到目标位置 → 释放吸盘
ArtObj关节物体操作打开/关闭抽屉、冰箱门等（通过 marker 交互点控制关节角度）
ResetArm手臂复位脚本化技能：将手臂恢复到初始位姿（不涉及神经网络）
Wait等待脚本化技能：原地不动若干步
Noop透传不改变动作空间，将底层动作原样传递（用于不需要技能包装的场景）
HRL 训练配置关键参数
# config: rearrange/rl_hierarchical.yaml
habitat_baselines:
  trainer_name: ddppo          # DD-PPO 做分布式采样
  updater_name: HRLPPO          # 只对 HL 决策步计算 loss
  rollout_storage_name: HrlRolloutStorage  # 支持 skill 边界的特殊 buffer
  rl:
    policy:
      name: HierarchicalPolicy   # 顶层策略，管理 HL + LL 技能组
      hierarchical_policy:
        high_level_policy:
          name: NeuralHighLevelPolicy  # 可学习 HL 或 FixedHighLevelPolicy
        defined_skills: nn_skills      # 使用预训练 + 微调的技能集

10b. VER 异步分布式训练

Variable Experience Rollout — 将环境步进、推理、学习解耦为独立进程，通过共享内存通信。 每个环境收集可变长度的经验，用重要性采样 (IS) 修正采样偏差。

VER 四进程架构

1. Environment Workers（每个 env 一个进程）：执行 simulator step，将 (obs, reward, done) 写入 CPU 共享内存
2. Inference Workers（可配数量）：轮询共享内存，批量运行 policy forward pass，将 action 写回，将完整 experience tuples 写入 GPU 共享内存
3. Learner（主进程）：当 rollout buffer 满时，计算 GAE → 生成 mini-batches → PPO update → 复制最新权重到共享张量
4. Report Worker（独立进程）：异步接收指标写入 TensorBoard/WandB，不阻塞训练

Variable-Length Rollouts 的代价与修正

在标准 PPO 中，N 个环境每个采集固定 T 步，产生 (T, N, ...) 张量。最慢的环境拖慢所有人。 VER 让每个环境在固定时间窗口内采集尽可能多的步数 — 快环境贡献更多步，慢环境贡献更少步。 这导致每个环境对 loss 的贡献不均，因此 VER 用 重要性采样系数 (T+1) / samples_per_env 修正采样偏差。 每步经验还打上 policy_version 标签，stale steps 保留旧的 return 值，防止使用过期的 value 估计。

维度PPODD-PPOVER
并行方式单机多 env（VectorEnv）多 GPU 数据并行 + 梯度 all-reduce多进程异步：env / inference / learn 解耦
每 env 步数固定 T 步固定 T 步（preemption 后截断）可变步数（时间窗口内尽可能多）
瓶颈CPU 核心数 × env 复杂度慢 GPU worker（straggler）共享内存带宽 + inference worker 吞吐
同步机制barrier（等所有 env 完成）sync_frac preemption无 barrier — 完全异步
配置复杂度低中低 — 只需 trainer_name: ver
适用场景开发调试、小规模实验单机多卡、需要线性加速大规模异步训练、env 步长时间不均
如何选择训练方法

单卡开发 → PPO（配置最简单，TensorBoard 反馈最快）
单机多卡加速 → DD-PPO（几乎线性加速，与 PPO 同配置，改 trainer_name 即可）
大规模集群 / env 步长不均 → VER（异步架构，解耦 env/inference/learn）
长周期复合任务 → HRL（两级策略 + PDDL 规划，适用于 Rearrange 系列）
以上方法可组合使用：DD-PPO 与 HRL 兼容（updater_name: HRLPPO），VER 也与 HRL 兼容。
