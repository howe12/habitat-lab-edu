第8章 · 部署轨 — 从仿真到真机

真机部署：VLM 端到端

视觉语言模型（VLM）直接接收 RGB 图像 + 自然语言指令 → 输出控制命令。 无需地图、无需 ROS 导航栈、无需预建环境模型——一个模型解决全部问题。

本章概览：方案一（VLN+ROS）+ 方案二（VLM端到端）

从理解 VLM 导航原理，到选型、部署、与方法一对比。方法二仍在快速发展中——这里的方案基于 2024–2025 的 SOTA 工作。

两种真机方案：VLN→ROS Nav2（稳妥）和 VLM 端到端（前沿）。前置：第7章 VLN 和 第6章 RL训练。

方案一 · VLN → ROS Nav2

将第7章训练好的 VLN 模型部署到真实机器人：VLN 做高层决策 → 输出子目标位姿 → ROS Nav2 执行底层导航。

方案二 · VLM 端到端

不使用导航栈，视觉语言模型直接输出控制指令——无地图、端到端的 VLN 方案。

8-7 · 理解部署架构

① 案例含义 — 为什么用 VLN + Nav2 而不是端到端？

方法一的核心理念是分而治之：VLN 模型擅于理解指令 + 识别视觉地标（"走到厨房水池前"）， 但不擅于底层的避障和路径规划。ROS Nav2 正好相反——它能在已知地图中高效规划路径、动态避障， 但它不理解自然语言指令。

两者分工：VLN 负责"要去哪里"，Nav2 负责"怎么过去"。

💡 为什么中间加一个 goal_publisher_node？

VLN 模型输出的子目标可能只是 (x, y, θ) 三个数，但 Nav2 的 NavigateToPose action 需要完整的 geometry_msgs/PoseStamped。 goal_publisher_node 专门负责格式转换 + 坐标系变换（从 VLN 的局部坐标系到 map 坐标系）， 保持 vln_inference_node 干净——它只做一件事：模型推理。

✅ 这个架构的优点

• VLN 模型和 Nav2 独立运行，一方崩溃不影响另一方
• Nav2 已有成熟的定位/规划/避障能力，不需要从头实现
• 更换 VLN 模型（如换更大的 backbone）不影响导航层
• 调试时可以单独测试每个节点的输入输出

⚠️ 这个架构的局限

• 需要预建地图（Nav2 的依赖）——不能在新环境中即开即用
• VLN 和 Nav2 的目标粒度需要调校（子目标设太远则 VLN 失去控制
  子目标设太近则效率低）
• 多了一个通信环节，增加了延迟

vln_inference_node.py — 核心推理节点

#!/usr/bin/env python3 # vln_inference_node.py — VLN 模型推理节点 # 从 /camera/rgb 读图像 + 从参数服务器读指令 → 输出子目标位姿 import rospy import numpy as np import torch from sensor_msgs.msg import Image from geometry_msgs.msg import Pose2D from cv_bridge import CvBridge class VLNInferenceNode: def __init__(self): rospy.init_node("vln_inference_node") # 加载模型 model_path = rospy.get_param("~model_path", "models/vln_inference.pth") self.model = self._load_model(model_path) self.bridge = CvBridge() self.instruction = rospy.get_param("~instruction", "") # Publisher: 输出子目标位姿 (相对坐标) self.goal_pub = rospy.Publisher("/vln/subgoal_pose", Pose2D, queue_size=1) # Subscriber: 相机图像 camera_topic = rospy.get_param("~camera_topic", "/camera/rgb/image_raw") self.img_sub = rospy.Subscriber(camera_topic, Image, self._image_cb) rospy.loginfo("VLN 推理节点已启动，等待图像...") def _load_model(self, path): # 从 第6章 导出的 checkpoint 加载模型 # 实际使用时替换为你的 VLN Policy 类 model = torch.load(path, map_location="cpu") model.eval() return model def _image_cb(self, msg): # 收到图像 → 推理 → 发布子目标 cv_img = self.bridge.imgmsg_to_cv2(msg, "rgb8") # 预处理: resize → normalize → tensor img_tensor = torch.from_numpy(cv_img).permute(2, 0, 1).float() / 255.0 img_tensor = img_tensor.unsqueeze(0) # (1, 3, H, W) with torch.no_grad(): # 模型输出: 动作类别 (0=STOP, 1=FORWARD, 2=LEFT, 3=RIGHT) action_logits = self.model(img_tensor, self.instruction) action = torch.argmax(action_logits, dim=-1).item() # 将动作转换为子目标位姿 pose = self._action_to_pose(action) self.goal_pub.publish(pose) def _action_to_pose(self, action): # 简化版：将离散动作映射为相对位姿 # 0=STOP → 原地，1=FORWARD → 前方 1m，2=LEFT → 原地左转90° # 3=RIGHT → 原地右转90° pose = Pose2D() if action == 0: # STOP — 已到达最终目标 pose.x, pose.y, pose.theta = 0.0, 0.0, 0.0 elif action == 1: # MOVE_FORWARD pose.x, pose.y, pose.theta = 1.0, 0.0, 0.0 elif action == 2: # TURN_LEFT pose.x, pose.y, pose.theta = 0.0, 0.0, 1.5708 # 90° in rad elif action == 3: # TURN_RIGHT pose.x, pose.y, pose.theta = 0.0, 0.0, -1.5708 return pose if __name__ == "__main__": VLNInferenceNode() rospy.spin()

goal_publisher_node.py — 子目标→Nav2

#!/usr/bin/env python3 # goal_publisher_node.py — 将 VLN 输出的相对位姿转为 Nav2 Action Goal import rospy import tf2_ros import actionlib from geometry_msgs.msg import Pose2D, PoseStamped from move_base_msgs.msg import MoveBaseAction, MoveBaseGoal class GoalPublisherNode: def __init__(self): rospy.init_node("goal_publisher_node") # TF 缓存 — 用于把 VLN 输出的相对坐标转换到 map 坐标系 self.tf_buffer = tf2_ros.Buffer() self.tf_listener = tf2_ros.TransformListener(self.tf_buffer) # Nav2 / move_base Action 客户端 self.ac = actionlib.SimpleActionClient("move_base", MoveBaseAction) rospy.loginfo("等待 move_base action server...") self.ac.wait_for_server() rospy.loginfo("已连接到 move_base!") # 订阅 VLN 子目标 rospy.Subscriber("/vln/subgoal_pose", Pose2D, self._goal_cb) def _goal_cb(self, pose2d): if pose2d.x == 0 and pose2d.y == 0 and pose2d.theta == 0: rospy.loginfo("收到 STOP 信号，取消导航") self.ac.cancel_all_goals() return # 根据当前 odom 变换，将相对坐标转为绝对 map 坐标 try: trans = self.tf_buffer.lookup_transform( "map", "base_link", rospy.Time(0), rospy.Duration(1.0) ) except Exception as e: rospy.logwarn(f"TF lookup 失败: {e}") return goal = MoveBaseGoal() goal.target_pose.header.frame_id = "map" goal.target_pose.header.stamp = rospy.Time.now() # 在当前位置基础上叠加 VLN 输出的相对位移 goal.target_pose.pose.position.x = trans.transform.translation.x + pose2d.x goal.target_pose.pose.position.y = trans.transform.translation.y + pose2d.y goal.target_pose.pose.orientation.w = 1.0 # 保持当前朝向 rospy.loginfo(f"发送导航目标: ({goal.target_pose.pose.position.x:.2f}, " f"{goal.target_pose.pose.position.y:.2f})") self.ac.send_goal(goal) if __name__ == "__main__": GoalPublisherNode() rospy.spin()

💡 先 Mock 测试，再接入真实 VLN 模型

建议先用一个 mock 节点替代 vln_inference_node（例如：每 3 秒发布一个固定的子目标坐标）， 先验证整个 pipeline 的通信链路（RGB → ??? → subgoal → Nav2 → /cmd_vel → 机器人移动）可以跑通。

通信链路验证通过后，再替换为真实的 VLN 推理节点。这样可以分隔问题域——如果导航失败， 你知道是 VLN 输出的问题还是 Nav2 配置的问题。

⚠️ 适配时的关键检查项

1. 相机内参：RealSense D435 的内参与 Habitat 仿真中的针孔相机参数不同。 如果 VLN 模型在仿真中训练时使用了特定的 FOV/分辨率，真机采集的图像需要做预处理匹配（crop/resize/FOV 矫正）。
2. TF 树完整性：确保 map → odom → base_link → camera_link 链完整， 否则 goal_publisher_node 的 TF lookup 会失败。
3. Nav2 地图格式：预建地图需要与当前环境一致。如果环境布局变了，需要重新建图。
4. Python 版本兼容：ROS 1 (Python 3.8) 和 ROS 2 (Python 3.10) 的 pickle 格式兼容——确保 第6章 导出的 checkpoint 能在两边加载。

④ 运行效果验证方法

验证步骤命令/方法预期结果
1. 节点都在 rosnode list 或 ros2 node list 看到 vln_inference, goal_publisher, move_base/bt_navigator
2. Topic 有数据 rostopic echo /vln/subgoal_pose 每收到一帧图像后，输出一个子目标位姿
3. Nav2 收到 goal 观察 rviz 中的 /plan 和 /global_plan 每次 VLN 输出新子目标时，Nav2 重新规划路径
4. 机器人移动 直接观察机器人 机器人按规划路径移动，到达子目标后停止等待下一个
5. STOP 判定 VLN 输出 STOP 后 rostopic echo /move_base/status goal 被取消，机器人停止
⚠️ 常见调试场景

场景 1："机器人不动" → 检查 rostopic echo /cmd_vel 是否有输出。 如果 /cmd_vel 为空 → Nav2 没收到 goal → 检查 goal_publisher_node 的 TF lookup 是否成功。
场景 2："机器人走错方向" → 90% 是坐标系问题。VLN 输出的坐标是相对 base_link 还是 map？ 在 rviz 中添加 Pose 显示，可视化每个子目标的位置。
场景 3："VLN 推理太慢" → 模型在 CPU 上推理可能 >500ms。考虑使用 ONNX Runtime 或 TensorRT 加速，或将模型量化到 FP16/INT8。

💡 调试建议：分阶段验证

Phase 1：不用 VLN 模型，用手动 rviz "2D Nav Goal" 验证 Nav2 基本导航正常
Phase 2：用 mock_vln_node（固定坐标输出）验证 vln → goal_publisher → Nav2 链路
Phase 3：接入真实 VLN 模型，先用 rosbag 录好的图像数据做离线推理测试
Phase 4：真机实时推理 + 导航

8-5 · 云-边-端 VLN 架构

① 案例含义

8-1 到 8-4 的架构假设 VLN 模型本地运行。但当模型大到 80 亿参数（JanusVLN）， 真机的 Jetson / 树莓派根本跑不动。8-5 介绍云-边-端分离方案： 真机负责感知+执行，云端 GPU 负责推理。

import rclpy, cv2, requests, json from sensor_msgs.msg import Image from cv_bridge import CvBridge class VLNCloudBridge(rclpy.node.Node): def __init__(self): super().__init__('vln_cloud_bridge') self.bridge = CvBridge() self.cloud_url = "http://L40_IP:8000/step" self.instruction = "Go to the kitchen" # 可动态更新 # 订阅相机 self.sub = self.create_subscription( Image, '/camera/color/image_raw', self.on_image, 10) # 发布子目标给 Nav2 self.goal_pub = self.create_publisher( PoseStamped, '/goal_pose', 10) def on_image(self, msg): # 1. ROS Image → JPEG bytes cv_img = self.bridge.imgmsg_to_cv2(msg, 'bgr8') _, jpg = cv2.imencode('.jpg', cv_img, [cv2.IMWRITE_JPEG_QUALITY, 80]) # 2. 上传到云端 resp = requests.post(self.cloud_url, files={'image': jpg.tobytes()}, data={'instruction': self.instruction}, timeout=5) # 3. 解析动作 action = resp.json()['action'] # "MOVE_FORWARD" self.get_logger().info(f'Action: {action}') # 4. 映射为 Nav2 目标并发布 if action == 'STOP': self.cancel_nav() else: goal = self.action_to_goal(action) self.goal_pub.publish(goal)

④ 每步延迟预算

阶段耗时占比优化空间

⚠️ 推理速度是瓶颈——但不是死结

2 秒/步听起来慢，但 VLN 导航通常 10-30 步到达目标 → 总耗时 25-75 秒，实际可用。 如果接受不了，可以：模型量化到 INT8（速度 2x）、换小模型（Qwen2.5-VL-3B）、 或等下一代 GPU（RTX 5090 推理 <0.5s）。

8-7 · 理解 VLM 导航原理

① 案例含义 — VLM 导航与传统 VLN 有什么不同？

传统 VLN（如 第6章 训练的模型）是一个任务专用的小模型：CNN 编码图像 + LSTM 编码指令 → 输出离散动作。 VLM 导航则使用一个通用的大模型（数亿到数十亿参数），已在海量图文数据上预训练过， 具备对场景、物体、空间关系的广泛理解。

核心区别：VLN 需要针对导航任务专门训练；VLM 已经"知道"什么是厨房、走廊、门口， 导航只是它推理能力的延伸。

② 核心论文思路拆解

📄 NaVid-4D (CVPR 2025)

核心思路：将导航形式化为视频预测任务。
输入：历史视频帧（多帧 RGB）+ 指令文本
输出：未来轨迹（一系列 waypoint 坐标）
特点：不显式建图，利用视频中的时序信息隐式编码空间结构。
模型：基于 Video Diffusion 架构，在 Habitat 仿真中训练。
与我们的关系：可以复用 第6章 训练出的模型作为特征提取器。

📄 NaVILA (CoRL 2024)

核心思路：结合 VLM 的语义理解 + 传统导航策略。
输入：RGB 图像 + 指令
输出：子目标（不是最终动作，而是中间目标点）
定位：介于方法一和方法二之间——VLM 做高层规划，但还需要底层控制器。
与我们的关系：可视为方法一的升级版：将 VLN 模型替换为大 VLM。

🔑 关键洞察：VLM 导航的本质是 "视觉问答 + 行动"

你可以把 VLM 导航理解为：

Q（指令）："走到厨房水池前面"
VLM 看当前图像：识别到 "我在客厅，前方是走廊，左边是厨房门"
VLM 推理："我应该往前走，到厨房门口再左转"
A（动作）：FORWARD

每一步都是一个新的视觉问答循环。VLM 不维护地图——它依赖实时视觉 + 常识推理来做决策。 这最接近人类导航的方式：你不需要心里有精确的 2D 栅格地图，你只需要"看到前方是什么"就能决定下一步。

8-8 · 模型选型与部署方案

① 案例含义

开源和商业 VLM 模型各有优劣。选型取决于你的硬件算力（Spark-I 的 i5-1240P 无独显 vs 是否有远程 GPU 服务器）、 延迟要求（实时导航需要 <500ms 推理）和费用预算。

② 模型对比

模型类型参数量推理速度 (CPU)导航能力硬件需求
Qwen2-VL-2B 开源 2B ~300ms (量化) 基础理解，可 fine-tune 4GB RAM，CPU 可行
LLaVA-1.6-7B 开源 7B ~2s (CPU) 较强推理，需量化 8GB VRAM (GPU)，14GB RAM (CPU)
NaVid-4D 研究 (复现) ~1B ~200ms (GPU) 专为导航优化 4GB VRAM
GPT-4V (API) 商业 API 未公开 ~500ms–2s (网络) 极强常识推理 仅需网络连接
Claude Vision (API) 商业 API 未公开 ~300ms–1s (网络) 极强空间推理 仅需网络连接
Qwen2-VL-72B (API) 商业/开源 72B ~2–5s (API) 最强综合能力 仅需网络连接

🖥️ 方案 A：本地推理（Spark-I 离线运行）

选型：Qwen2-VL-2B 量化版 (INT8/INT4)
优点：无网络依赖、低延迟、数据不出设备
缺点：导航能力弱于大模型，可能在复杂场景出错
适配：使用 ONNX Runtime 或 llama.cpp 在 i5-1240P 上运行
调优：可以用 R2R 数据做少量 fine-tune 提升导航表现

☁️ 方案 B：云端 API（机器人连 WiFi）

选型：GPT-4V 或 Claude Vision API
优点：最强推理能力，几乎不需要 fine-tune
缺点：依赖网络、有延迟波动、API 调用费用
适配：ROS 节点通过 HTTP/gRPC 调用 API
适用场景：研究原型验证、Demo、非实时任务

💡 推荐策略：本地 + 云端混合

先用方案 B（云端 API）快速验证 VLM 导航的可行性——用最聪明的模型看看你的场景是否能靠纯视觉+语言解决。 确认可行后，再考虑方案 A 部署本地模型以获得更低的延迟和更稳定的表现。

Spark-I 的 i5-1240P 虽然无独显，但 16GB RAM + Intel Iris Xe 集成显卡可以跑 ONNX 量化模型。 Leo 如果有更强算力（或外接 GPU），选择余地更大。

8-9 · 真机 Pipeline 实现

① 案例含义

实现一个极简的 ROS 节点：订阅相机图像 → 调用 VLM → 解析动作 → 发布 /cmd_vel。 与方法一的五节点架构不同，方法二的整个 pipeline 只有一个核心节点。

② 核心代码

#!/usr/bin/env python3 # vlm_navigation_node.py — VLM 端到端导航（云端 API 方案） # 设计原则：极简单节点，低耦合 import rospy import base64 import requests from sensor_msgs.msg import Image from geometry_msgs.msg import Twist from cv_bridge import CvBridge import cv2 class VLMNavigationNode: def __init__(self): rospy.init_node("vlm_navigation") # 配置 self.instruction = rospy.get_param("~instruction", "Navigate to the kitchen and stop in front of the sink.") self.api_key = rospy.get_param("~api_key", "") self.api_url = rospy.get_param("~api_url", "https://api.openai.com/v1/chat/completions") self.bridge = CvBridge() self.cmd_pub = rospy.Publisher("/cmd_vel", Twist, queue_size=1) # 限速：每秒最多推理一次 self._last_inference = rospy.Time.now() self._inference_interval = rospy.Duration(1.0) rospy.Subscriber("/camera/color/image_raw", Image, self._image_cb) rospy.loginfo("VLM 导航节点已启动") def _image_cb(self, msg): if rospy.Time.now() - self._last_inference < self._inference_interval: return # 限速 self._last_inference = rospy.Time.now() # 1. 图像 → Base64 cv_img = self.bridge.imgmsg_to_cv2(msg, "rgb8") _, buffer = cv2.imencode(".jpg", cv_img, [cv2.IMWRITE_JPEG_QUALITY, 70]) img_b64 = base64.b64encode(buffer).decode() # 2. 调用 VLM API action = self._call_vlm_api(img_b64) # 3. 动作 → /cmd_vel twist = self._action_to_twist(action) self.cmd_pub.publish(twist) def _call_vlm_api(self, img_b64): # GPT-4V / Claude API 调用 headers = {"Authorization": f"Bearer {self.api_key}", "Content-Type": "application/json"} prompt = f"""You are a robot navigation assistant. Current instruction: {self.instruction} Based on the current camera view, output ONE action from: [FORWARD, TURN_LEFT, TURN_RIGHT, STOP] Output ONLY the action name, nothing else.""" payload = { "model": "gpt-4o", "messages": [{ "role": "user", "content": [ {"type": "text", "text": prompt}, {"type": "image_url", "image_url": {"url": f"data:image/jpeg;base64,{img_b64}"}} ] }], "max_tokens": 10, "temperature": 0.0, } resp = requests.post(self.api_url, headers=headers, json=payload, timeout=5) action = resp.json()["choices"][0]["message"]["content"].strip() return action def _action_to_twist(self, action): twist = Twist() if action == "FORWARD": twist.linear.x = 0.3 # 前进 0.3 m/s elif action == "TURN_LEFT": twist.angular.z = 0.8 # 左转 0.8 rad/s elif action == "TURN_RIGHT": twist.angular.z = -0.8 # 右转 0.8 rad/s elif action == "STOP": pass # 全零 = 停止 else: rospy.logwarn(f"未知动作: {action}") return twist if __name__ == "__main__": VLMNavigationNode() rospy.spin()

# 本地推理变体 — 使用 Qwen2-VL-2B 量化版 + ONNX Runtime # 替换上面 _call_vlm_api() 方法： def _call_local_vlm(self, cv_img): import onnxruntime as ort import numpy as np # 预处理：resize 到模型输入尺寸 img = cv2.resize(cv_img, (384, 384)) img = img.astype(np.float32) / 255.0 img = np.transpose(img, (2, 0, 1))[np.newaxis, ...] # ONNX 推理 logits = self.ort_session.run(["logits"], {"image": img})[0] action = ["STOP", "FORWARD", "TURN_LEFT", "TURN_RIGHT"][np.argmax(logits)] return action
⚠️ VLM 端到端导航的关键挑战

1. 延迟：云端 API 调用可能 500ms–2s，导致机器人每 2 秒才能做一次决策。高速行进时这很危险。 降低分辨率（320×240）和使用 JPEG 压缩可减少传输时间。
2. 安全性：VLM 可能输出不合理动作（如前方是墙却说 FORWARD）。建议在 /cmd_vel 输出前加 一个安全屏障节点（基于激光雷达的碰撞检测），拦截危险指令。
3. 累积误差：每个动作的小误差会累积。需要定期让 VLM "重新审视"当前位置与目标的偏差。
4. 提示工程：prompt 的设计直接影响导航质量。你可能需要针对你的环境反复调试 prompt。

💡 安全屏障：必须加

无论选择方法一还是方法二，在 /cmd_vel 前都应加一个碰撞检测节点： 订阅 /scan（激光雷达），如果前方 < 0.3m 内有障碍物，强制发布 STOP（Twist() 全零）， 覆盖上游所有节点（VLN 或 VLM）的输出。Spark-I 的 360° LiDAR 为此提供了很好的硬件基础。

🎯 决策指南

选方法一，如果：
• 你的工作场景是固定的（办公室/家庭楼层布局不变）
• 你重视安全性和路径最优性
• 你的团队有 ROS 经验
• 你需要在 Spark-I / Leo 上低延迟运行

选方法二，如果：
• 你需要快速验证 VLN 的概念而不想配置导航栈
• 你的使用场景不固定（多变的临时环境）
• 你更偏向 CV/ML 背景而非 ROS 背景
• 你可以接受较高的 API 费用和网络延迟
• 你想探索最前沿的 VLM 能力边界

最优方案：两者结合。用方法二做全局语义规划（VLM 看全景图 → "先去厨房，再去卧室"）， 用方法一做局部路径执行（Nav2 负责每个段落的安全导航）。
